Depop Scraper — Architecture Overview

Vintage-hat resale-arbitrage pipeline · pure Python stdlib · ~/Dev/depop-scraper

1 reqper page of 24 listings — full data, no detail fetch
~90%+decided by rules, $0 tokens
0 depsstdlib only (urllib + sqlite3)
0 loginfresh UUID headers, no auth

One sentence: poll Depop's private search API for cheap vintage hats, score each listing against a hand-built value rubric, escalate only the ambiguous ones to an LLM, and push a phone notification when something is worth buying to resell — without ever notifying twice.

Data flow & key breakpoints

1
depop.py · search()

Fetch — private "presentation" API

GET /presentation/api/v1/search/products/ with fresh UUID headers, paginated via the page_info.last cursor.

Why this endpoint: the search row already contains full description, brand, condition & images — so no per-listing detail request is needed. Official webapi.depop.com v2/v3 are deprecated; browser automation would be heavier and bannable.

Breakpoint: fresh depop-device-id / session-id / search-id UUIDs per request — without them the API returns 400. No login/session is needed.

2
depop.py · _get()

Rate-limit discipline

Jittered delay (delay + rand(0,1) + attempt*5) before every call; retry transient network errors only.

Breakpoint: on 403 / 429 it raises RateLimited and aborts the whole scrape — it does not retry. Politeness over completeness: better to skip a run than get the device fingerprint banned.

3
store.py · is_seen()

Dedup against SQLite

Drop any listing id already in seen.db before doing any scoring work.

Why early: dedup before normalize/score/LLM means zero tokens and zero work spent on listings already processed. The DB is the idempotency boundary.

4
depop.py · normalize()

Normalize raw row → pipeline schema

Flatten pricing (pricing[final_price_key].total_price), pull first image per format, derive title/seller from the slug.

Why: decouples the messy API shape from scoring. A price/range guard (0 < price ≤ max_price) drops junk before scoring.

5
scoring.py · value_rubric.json

Rule-based scoring (the brain)

Keyword/regex points: brand tier, era, defunct teams, 3+ rarity factors, negatives, and per-brand price gating. Clamp 0–100 → buy ≥70 / maybe ≥60 / skip.

Why rules first: ~90%+ of listings get a confident verdict for $0. The expensive LLM is reserved for the narrow uncertain band only.

Breakpoint: per-brand max_buy thresholds + the "3+ rarity factor" rule come straight from the human arbitrage playbook — domain knowledge encoded as data, not code.

6
llm.py · judge_maybes()

LLM fallback — "maybe" band only

One batched Haiku call for all maybes. Text for each; image attached only for the top N by score (vision is pricey). System prompt is cache_control: ephemeral.

Why batched + capped: tokens are logged and capped per run. Prompt-caching the system prompt cuts cost across runs. Verdict flips maybe → buy/skip.

7
notify.py · ntfy.sh

Notify — buys above threshold

JSON publish to ntfy topic with click-action to the listing. Gated by verdict==buy && score≥90 && price≤max.

Why JSON publish: safe for non-ASCII titles. The notify gate (90) is stricter than the buy threshold (70) — buy is recorded, but you're only pinged for the strongest.

8
store.py · mark() / log_run()

Persist — mark seen, log run

INSERT OR IGNORE every scored listing (with notified flag); append run stats (listings/buys/notified/tokens).

Why: marking everything seen — not just buys — is what guarantees "never notify twice". The runs table gives cost/yield telemetry over time.

Scheduling: a one-shot run_once is the unit. --loop sleeps cadence_minutes between runs; a launchd plist (com.suvan.depop-scraper.plist) runs it headless. Crash-resilience comes from SQLite state, not from the loop staying alive.

Architectural decisions & the reason for each

DecisionWhy
Private presentation API, not official API or browser Official v2/v3 deprecated; presentation API returns full description/brand/condition/images in the search row → 1 request/page, no detail fetch, no headless browser.
Fresh UUIDs every request, no login API only needs well-formed device/session/search UUID headers (400 without). No auth state to manage or leak; each request looks independent.
Abort scrape on 403/429 (no retry) Protect the IP/fingerprint. A missed run self-heals next cadence; a ban doesn't.
Dedup in SQLite before scoring Idempotency + "never notify twice"; spends zero work/tokens on already-seen ids.
Rules-first, LLM only on the maybe band Cost control: ~90%+ decided for free; pay the model only where rules are genuinely uncertain.
Rubric as JSON data, not Python Domain playbook (brand max-buys, defunct teams, rarity factors) tunable without touching logic.
Vision on top-N maybes only + cached system prompt Images dominate token cost; cap them and cache the static instruction to keep $/run low.
Notify gate (90) stricter than buy gate (70) Record all buys for analysis; only interrupt the human for high-confidence finds.
Pure stdlib (urllib + sqlite3), zero deps Trivial to run under launchd/cron on any Mac; nothing to install, no venv drift.
State in DB, scheduling outside the process Each run is a pure function of (config, DB); loop/launchd just decides when. Crash-safe.
